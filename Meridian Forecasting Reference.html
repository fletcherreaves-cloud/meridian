1
Core Forecast Formula
DOW Model & Dialed-In Model — the primary calculation for most stores

Final Expression

forecast = lyAdjH × opsFactor × (1 + wAdj) × (1 + trendFactor) × (1 + eventFactor) × (1 + plusUp)

This formula runs inside forecastDay() for every store × date combination. Each factor is computed independently and is described in its own section below.

Complete Calculation Flow

#VariableDescriptionSee Section
1lyRawActual sales from same DOW ~52 weeks ago§7
2lyAdjLY blended with district avg when lyW < 1.0 (DI calibration)§17
3lyAdjHlyAdj × holidayLyAdj (corrects for holiday alignment)§9
4trendFactorClamped YOY trend blend (±15% max)§8
5opsFactorOEPE / TPPH / staffing adjustment§10
6wAdjWeather impact adjustment (empirical or sliders)§11
7eventFactorLearned historical event impact§12
8plusUpManual percentage adjustment§13
When Core Formula Does NOT Run When a store is assigned the AE (Adaptive Ensemble) or EWMA model, those models short-circuit before the Core Formula executes. The Core Formula runs for DOW and DI model assignments only.

3
EWMA DOW
Exponentially Weighted Moving Average — same day-of-week
forecast = Σ( α × (1−α)^i × S_i ) / Σ( α × (1−α)^i ) where: S_i = actual sales i weeks ago on the same day-of-week i = 0 (most recent), 1, 2, ... up to 13 (oldest, 14 weeks max) α = 0.25 (standalone) | 0.22 (AE weekday) | 0.30 (AE weekend)

Decay Weight Table (α = 0.25)

Weeks AgoWeightCumulative %
Most recent (i=0)0.250025.0%
1 week (i=1)0.187543.8%
2 weeks (i=2)0.140657.9%
3 weeks (i=3)0.105568.4%
4 weeks (i=4)0.079176.3%
5 weeks (i=5)0.059382.2%
6 weeks (i=6)0.044586.7%

Minimum required: 3 same-DOW data points. If fewer exist, EWMA returns null and the caller falls back to the LY model.

Responds 2–3× faster than a simple rolling average A 6-week simple average weights all weeks equally at 16.7%. EWMA with α=0.25 gives the most recent week 25% weight — nearly double — and the most recent 5 weeks account for 82% of the result. This makes EWMA ideal for stores in transition where the oldest data is misleading.
4
Adaptive Ensemble (AE)
5-signal blend — district avg 9.29% MAPE vs LifeLenz 9.51% (Sep 2025 – May 2026)

The Five Signals

Signal 1 — EWMA DOW

ewmaSignal = forecastEWMA(loc, date, α) α = 0.22 on weekdays | 0.30 on weekends

Signal 2 — LY Adjusted

# Find actual LY value (±3 day search for DOW match) lyVal = sales[date − 364 ± 3 days, same DOW] # Compute YOY adjustment from recent 90 days vs same 90-day window last year adj = avg(sales last 90d) / avg(sales 364d−274d ago) − 1 adj = clamp(adj, −0.20, +0.20) lySignal = lyVal × (1 + adj)

Signal 3 — Short-term Momentum

factor = avg(last 14d same-DOW actuals) / avg(14d–28d same-DOW actuals) factor = clamp(factor, 0.85, 1.15) momentumSignal = ewmaSignal × factor # requires ewmaSignal ≠ null

Signal 4 — Monthly Seasonality

thisMon = avg sales for this month, same DOW (requires ≥ 4 data points) allTime = avg sales for all-time, same DOW factor = clamp(thisMon / allTime, 0.80, 1.20) seasonalSignal = ewmaSignal × factor

Signal 5 — Adaptive DI

Per-store calibrated EWMA blend across 14d/28d/42d windows. See §5 for full detail.

5
Adaptive DI
Per-store calibrated blend of 14d / 28d / 42d EWMA windows — Signal 5 inside AE
EWMA(14d) = exponential average of same-DOW actuals within last 14 days, using alpha EWMA(28d) = exponential average of same-DOW actuals within last 28 days, using alpha EWMA(42d) = exponential average of same-DOW actuals within last 42 days, using alpha AdaptiveDI = (w2 × EWMA(14d) + w4 × EWMA(28d) + w6 × EWMA(42d)) / (w2 + w4 + w6) # renormalized over available windows only
ParameterRoleDefaultRange
w2Weight on 14-day EWMA window0.40calibrated per store
w4Weight on 28-day EWMA window0.35calibrated per store
w6Weight on 42-day EWMA window0.25calibrated per store
alphaEWMA decay factor within each window0.20calibrated per store

Per-store {w2, w4, w6, alpha} values are stored in the AE params table (mf_ae_params in localStorage) and updated on recalibration. A store with high w2 responds quickly to recent changes; high w6 relies on longer-term stability.

6
Multi-Model Engine
Four independent models run in parallel for backtest and comparison views

The forecastModels() function runs four models simultaneously. Used by Forecast Accuracy Report, DI Compare, and model selection backtesting.

KeyNameMethod
m1 Composite Core Forecast Formula with current model assignment (DOW/DI/AE/EWMA)
m2 Trend-Only lyAdj × (1 + wTrend) × (1 + plusUp) — no ops factor, no weather
m3 Momentum Exponential decay on 8 most-recent same-DOW actuals: Σ(e^{−0.3i} × S_i) / Σ(e^{−0.3i})
m4 Regression OLS linear regression on 12 most-recent same-DOW actuals; projects next value on trendline
ens Ensemble Equal-weight average of all available model forecasts

Momentum Model Detail

recent = last 8 same-DOW actuals, sorted newest-first momentum = Σ( e^{−0.3×i} × sales[i] ) / Σ( e^{−0.3×i} ) for i = 0..7

Regression Model Detail

pts = last 12 same-DOW actuals, sorted oldest-first xs = [0, 1, 2, ..., n−1] ys = [sales_0, sales_1, ..., sales_{n-1}] slope = Σ((x − x̄)(y − ȳ)) / Σ((x − x̄)²) intercept = ȳ − slope × x̄ regression = (intercept + slope × n) × (1 + plusUp) # projects at point n

Last Year Lookup (lyRaw)
Priority chain — skips holidays and tagged events automatically
Priority offsets (days back from forecast date): −364 # exact 52 weeks (same DOW guaranteed if year-length ≡ 0 mod 7) −357 # 51 weeks −371 # 53 weeks −378 # 54 weeks −350 # 50 weeks −385 # 55 weeks −343 # 49 weeks For each candidate: 1. Must be same day-of-week as forecast date 2. Must NOT be a holiday (auto-excluded) 3. Must NOT be a user-tagged event for this store 4. Must have actual sales > 0 lyRaw = first candidate meeting all conditions (or 0 if none found)
No Blending, No Dampening Meridian does NOT blend multiple LY candidates or apply outlier dampening. Each lyRaw value is a real historical sales day the user can verify. Synthetic blended values were removed (v4.x) because they broke user trust — the displayed LY didn't match any actual day in the data.
8
Trend Windows (t2 / t4 / t6)
YOY growth by recency band, blended into a single trendFactor
# Three recency bands — each measures YOY growth for same DOW in that week range t2 = avg YOY growth in weeks 1–2 back from latest actual, same DOW only t4 = avg YOY growth in weeks 3–4 back from latest actual, same DOW only t6 = avg YOY growth in weeks 5–6 back from latest actual, same DOW only # YOY growth for each data point point_i = (sales[week i] − sales[week i − 52 weeks]) / sales[week i − 52 weeks] # Global trend (weighted blend) globalTrend = t2 × tw.t2 + t4 × tw.t4 + t6 × tw.t6 (default: t2×0.50 + t4×0.30 + t6×0.20) # DOW-specific trend (this weekday's YOY over last N weeks) dowSpecific = avg YOY for THIS specific DOW over last weeksBack (6) weeks # Blend: 65% global + 35% DOW-specific (when dowSpecific ≥ 3 data points) blendedTrend = globalTrend × 0.65 + dowSpecific × 0.35 # Apply alpha compression and clamp to ±15% trendFactor = clamp(blendedTrend × 0.30, −0.15, +0.15)

Holiday Correction
Adjusts when holiday alignment differs between forecast date and its LY baseline
# Determine if forecast date and LY date (−364 days) are holidays todayIsHoliday = isHoliday(date) # from built-in holiday table ly364IsHoliday = isHoliday(date − 364) # excluded from lyRaw lookup? holidayLyAdj = if same holiday both sides → 1.0 # no adjustment needed if today is holiday, LY was not → getHolidayAdj(date, loc, history) # reduces LY (expect lower) if LY was holiday, today is not → 1 / max(0.30, getHolidayAdj(LY)) # LY understates normal volume if neither is a holiday → 1.0 lyAdjH = lyAdj × holidayLyAdj

getHolidayAdj() returns a per-holiday impact multiplier derived from the district's historical actuals vs the non-holiday DOW baseline. Holidays in the built-in table include: Christmas Eve/Day, Thanksgiving (Wed/Thu/Fri), Independence Day, New Year's Eve/Day, Labor Day, Memorial Day, and others.

10
Ops Factor
Adjusts forecast based on 6-week rolling operational performance
p = compute6wk(loc, ds, weeksBack) # 6-week rolling averages of all ops metrics t = targets for this store om = opsMults settings rawOpsF = 1.0 # OEPE adjustments (vs target or store historical avg when opsNorm enabled) if p.oepe > effOepe + 20 rawOpsF × = (1 + om.oepeSeverePenalty / 100) elif p.oepe > effOepe rawOpsF × = (1 + om.oepePenalty / 100) else rawOpsF × = (1 + om.oepeBonus / 100) # KVST (Kitchen Valid Service Time) if p.kvst > t.tKvst rawOpsF × = (1 + om.kvstPenalty / 100) # KVSU (Kitchen Valid Service Uptime) if p.kvsu < t.tKvsu rawOpsF × = (1 + om.kvsuPenalty / 100) # Parking % if p.park > t.tPark rawOpsF × = (1 + om.parkPenalty / 100) # TPPH if p.tpph < effTpph rawOpsF × = (1 + om.tpphPenalty / 100) else rawOpsF × = (1 + om.tpphBonus / 100) # Act vs Need (staffing alignment) if p.actVsNeed < −30 rawOpsF × = (1 + om.shortStaffPenalty / 100) elif p.actVsNeed > +15 rawOpsF × = (1 + om.readyStaffBonus / 100) rawOpsF = max(om.opsFactorFloor / 100, rawOpsF) opsFactor = 1 + (rawOpsF − 1) × calOpsMult # calOpsMult from DI calibration (0.0–1.5)
Ops Normalization Mode When "Ops Normalization" is enabled in settings, effOepe and effTpph are drawn from the store's own 6-week rolling historical average instead of its targets. This prevents penalizing stores that consistently operate at a different baseline from their target while still rewarding genuine improvement.
11
Weather Adjustment
Empirical per-store calibration via linear regression of historical sales vs weather

Mode 1 — Empirical (Preferred)

# Per-store linear regression: sales deviation from DOW baseline vs weather intensity baseline[dow] = avg sales on non-rainy days for this DOW # clean baseline dev = (actual_sales − baseline[dow]) / baseline[dow] # Regression: y = slope × x for each weather variable rainCoef = OLS slope of (rain_inches, dev) pairs where rain > 0.1" hotCoef = OLS slope of ((tmax−95)/10, dev) pairs where tmax > 95°F coldCoef = OLS slope of ((35−tmax)/35, dev) pairs where tmax < 35°F windCoef = OLS slope of ((wmax−25)/25, dev) pairs where wmax > 25 mph # Clamp coefficients to plausible range rainCoef = clamp(rainCoef, −0.15, +0.02) hotCoef = clamp(hotCoef, −0.10, +0.01) coldCoef = clamp(coldCoef, −0.08, +0.01) windCoef = clamp(windCoef, −0.05, +0.01) # Apply to forecast date's weather wAdj = rainCoef × min(rain, 3) + hotCoef × (tmax − 95) / 10 # only if tmax > 95 + coldCoef × (35 − tmax) / 35 # only if tmax < 35 + windCoef × (wmax − 25) / 25 # only if wmax > 25 wAdj = clamp(wAdj, −0.15, +0.03) # final clamp

Mode 2 — Settings Sliders (Fallback)

When empirical calibration is disabled or fewer than 4 samples exist per variable, weather adjustment uses user-configured percentage sliders from Settings (hotDay, coldDay, niceDay, lightRain, heavyRain, highWind, veryHighWind). Final result clamped at max −10% / +3%.

Forecast Weather Fetch

Future weather is fetched from Open-Meteo (no API key required) per store coordinates. Rate-limit handling: 1.1-second inter-store delay + progressive retry backoff on HTTP 429: 0s → 15s → 30s → 60s → 120s (5 attempts max).

13
Plus-Up
Manual percentage applied after all model calculations
# Priority chain — highest wins plusUp = if plusUpByStore[loc] defined → use store-level value elif any patch p contains loc and plusUpByPatch[p] defined → use patch-level value else → use global plusUp setting # Application (always the last multiplicative step) forecast = priorResult × (1 + plusUp / 100)

Applied in: Core Formula (DOW/DI), Trend-Only (m2), Momentum (m3), Regression (m4). Not applied in: AE model (which uses its own signal blend directly).

14
Guest Count Cross-Check
Validates sales forecast against an independent GC-based projection
# GC forecast (same LY priority chain as sales) gcLY = historical GC for same DOW ~52 weeks ago gcTrend = blended DOW trend from GC history (same t2/t4/t6 structure) gcForecast = round(gcLY × (1 + gcTrend)) # Implied average check from the sales forecast impliedCheck = salesForecast / gcForecast # Store's actual recent check norm (6-week rolling) normCheck = avg(sales / gc) over last 42 days with paired data # Divergence deviation = (impliedCheck − normCheck) / normCheck # Flags flag = 'alert' if |deviation| > 0.20 = 'watch' if |deviation| > 0.10 = null otherwise

Minimum requirements: 5 recent rows with both sales and GC, normCheck > $0.50, gcLY > 0.

Breakfast / Lunch / Dinner — requires 3 Peaks data

17
Dialed-In Calibration
Grid search over ~21,500 parameter combinations to minimize MAPE per store

The 5 Parameters

ParameterRangeRole
lyW0.50 – 1.00LY weight vs district DOW average (lower = less trust in store's own prior year)
t20.10 – 0.80Weight on 2-week YOY trend (higher = faster response to recent changes)
t40.05 – 0.60Weight on 4-week YOY trend
t60.05 – 0.60Weight on 6-week YOY trend (higher = more stable, slower to adapt)
opsMult0.00 – 1.50Ops influence multiplier (0 = ignore ops, 1.5 = amplify ops signal)

Constraint: t2 + t4 + t6 ≤ 0.95 (preserves some global trend influence in all combinations). ~21,500 valid combinations after constraint enforcement.

Evaluation Method

# For each parameter combination {lyW, t2, t4, t6, opsMult}: Run forecastDay() on last 42 days of actuals for this store using COMPLETE pipeline (holiday adj + trend + ops + weather + event + plusUp) mape = avg |actual − forecast| / actual × 100 best_params = combination with lowest mape over the evaluation window
Fixed in v4.195 Prior to v4.195, the grid search used a simplified standalone formula that omitted holiday adjustments, event factors, and the full trend pipeline. This meant the "best" combination found was not actually best against real forecasts. The grid search now runs the identical forecastDay() pipeline the live model uses, ensuring calibrated params generalize correctly.

When to Recalibrate

  • Any settings change (targets, ops multipliers, weather settings)
  • After significant operational changes at a store (new GM, remodel, hours change)
  • After > 42 days since last calibration run
  • When MAPE drift status is "recalibrate" (§20)

MAPE — Mean Absolute Percentage Error
Primary forecast accuracy metric — lower is better
MAPE = ( 1/n ) × Σ | (actual_i − forecast_i) / actual_i | × 100
MAPERatingGuidance
< 5%ExcellentSchedule aggressively to model — high confidence
5–10%GoodReliable for planning; add small buffer for scheduling
10–15%FairReview model before locking. Check for anomalies or events.
15–25%PoorUse as starting point only; manual review required
> 25%Do Not TrustStructural model failure — insufficient or erratic data

Meridian reports MAPE at four windows: 6W, 4W, 2W, and 1W. The 6W window is the primary reference for scheduling decisions. Comparing 2W vs 6W reveals drift (see §20).

20
MAPE Drift Detection
Early warning that a model is getting worse relative to its longer-term baseline
mape2w = avg |actual − forecast| / actual × 100 over last 14 days mape6w = avg |actual − forecast| / actual × 100 over last 42 days drift = |mape2w − mape6w| status = 'ok' if drift < 2.0 = 'warn' if 2.0 ≤ drift < 5.0 = 'recalibrate' if drift ≥ 5.0
StatusDriftAction
OK< 2%No action
Warn2–5%Monitor; consider recalibrating
Recalibrate≥ 5%Run Dialed-In now — model is degrading

Common causes: new GM, competitor opening, remodel, operational hours change, seasonal shift not in LY. Usually resolves in 4–6 weeks as the change stabilizes, but recalibrating immediately locks the new baseline.

21
Confidence Intervals (σ)
Statistical uncertainty bands from the last 8 weeks of forecast errors
# Collect percentage errors over last 8 weeks (min 7 data points, |error| ≤ 40% only) errors = [(actual_i − forecast_i) / actual_i for each day in last 8 weeks] mean = avg(errors) σ = sqrt( Σ(error_i − mean)² / n ) # Confidence bands (±1σ ≈ 68% of outcomes) lower = forecast × (1 − σ) upper = forecast × (1 + σ)
σ ValueInterpretation
< 5%Tight — schedule confidently
5–10%Moderate — add small scheduling buffer
10–15%Wide — review before committing schedule
> 15%Very wide — use manual judgment

LifeLenz Bridge
WFM bias detection and adjusted projections for the schedule lock workflow

Historical Bias

# For each store, analyze last 12 weeks of days where projSales > 0 # (requires Labor Analysis loaded with "Projected Sales" / "WFM Projected Sales" column) biasPct = avg[ (actual − lfzProjection) / actual ] × 100 # Positive = LifeLenz under-forecasted (actual > projection) # Negative = LifeLenz over-forecasted (actual < projection) # Also computed per day-of-week: dowBias[d] = avg[ (actual − lfzProjection) / actual ] for rows with date.getDay() = d

Forward Adjustment (14 days)

# For each of the next 14 days: # Option A: Direct row exists in Labor Analysis for this date adjustedProjection = projSales × (1 − biasPct / 100) source = 'direct' # Option B: No direct row — apply DOW bias to Meridian forecast dowStat = dowBias[date.getDay()] adjustedProjection = meridianForecast × (1 + dowStat / 100) source = 'pattern'

The adjusted projection is the number to type into LifeLenz before schedule locks to produce a correctly calibrated labor schedule.
